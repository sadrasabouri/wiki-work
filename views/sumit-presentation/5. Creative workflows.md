# Creative Workflows for Presentation Generation

## The Workflows

**1. Template + fill**
- Define a template with `## Section` headings and `> fill:` prompts specifying what each slide should contain.
- Run it against the KB. The template encodes the argument structure; the fill prompts encode what evidence to pull.

Example: [[templates/sumit-lightning.md]] — each `> fill:` prompt specifies the audience, the source, and the claim to land.

**2. FlashFill convergence**
- Start from the desired output shape.
- Generate a first draft.
- Revise it
- Feed the revision back as a new input-output example.
- Repeat until the agent converges on the right framing.

Sumit used this for his Thursday pre-read: "each revision is another example, narrowing the hypothesis space." The skill is accumulating examples rather than specifying the methodology upfront. *[[Sadra-6May2026-Notes]]*

**3. Grounded examples first**
- Before building the structure, query the KB for the best concrete stories and examples.
- Let the examples drive the slide structure, not the other way around.

Sumit's explicit request: "3 concrete wiki examples for different audiences." The [[views/presentation-grounded-examples|grounded-examples presentation]] was generated this way — examples first, structure second. *[[Sadra-5May2026-TRANSCRIPT2]]*

**4. Audience-aware storytelling**
- Hold the slide structure constant;
- vary the story selection per audience.

The same KB contains stories relevant to engineers, managers, and executives — the skill picks different ones depending on `{{audience}}`. The template parameterizes the audience; the skill knows to match example specificity and vocabulary to it.
  
Example: [[templates/sumit-lightning.md]] uses `{{audience}}` to steer which failure stories to surface.

**5. Argument-first / claim-then-evidence**
- Start with the thesis you want to land.
- Query the KB for evidence, counter-evidence, and tensions that support or complicate each claim.
- Build slides as claim → evidence → tension.

**6. Self-referential anchoring**
- Use the presentation itself as the first example of the thing it describes.
- "This deck is a live view — it was generated from the KB you're hearing about."
- Every slide has a pointer back to the system producing it.

Sumit identified this as the strongest opener: "this deck is already feature one running." Requires `self_referential: true` in the template frontmatter. *[[self-referential-kb]], [[Sadra-5May2026-TRANSCRIPT2]]*

**7. Contradiction-surfacing**
- Query the KB for tensions and unresolved questions.
- Use them as narrative hooks — "here's what we don't know yet" is often more compelling than "here's what we proved."
- Pairs with open questions as the `ingest` skill per slide. Best for research audiences who distrust oversimplification.

**8. Reverse-engineering from conclusion**
- Decide the final slide's takeaway first.
- Work backward: what must the audience believe by slide N-1 to accept that conclusion? What by slide N-2?
- Build the template bottom-up from the desired landing point, then fill each hole forward.

Used implicitly in Sumit's pre-read: "I started from the content outline and had an agent fill each section." *[[2026-05-06]]*
