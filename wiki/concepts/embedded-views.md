# Embedded Views

A design option for how [[views|Views]] are rendered: instead of generating standalone output files via CLI, views are **embedded inside existing applications** (Word, PowerPoint, Excel, Notion) as prompt-powered "holes" in a document template.

Proposed by [[sumit-gulwani|Sumit]] in [[Sadra-3May2026-TRANSCRIPT|the May 3 sync]] as Option 2, contrasted with Option 1 (the current approach).

## Option 1 vs. Option 2

| | Option 1 (current) | Option 2 (embedded) |
|---|---|---|
| Output location | Standalone `.md` files generated by Claude Code CLI | Inside existing applications (Word, PPT, Excel) |
| Container | The CLI / rendered wiki | The application's native document format |
| "All applications are dead" | Yes — the wiki replaces them | No — existing apps are the delivery surface |
| Each field | Prompt generates the whole document | Each field/cell/section has its own prompt |

## How Embedded Views Work

In Option 2, a view definition is a document template with **prompt-powered holes**. For a presentation:
- The title slide's title = a prompt asking for a 6-word title for the project
- The collaborators field = a prompt that reads the wiki and lists contributors
- The speaker name = hard-coded (not a prompt)

The document has two modes: **editing mode** (you see the prompts) and **rendered mode** (you see the outputs). Refreshing the document recomputes all prompt-powered fields against the current wiki.

## Why This Matters

Embedded views reframe the KB system as infrastructure for existing applications rather than a replacement. The KB provides the knowledge layer; applications provide the surface. This is a more conservative positioning for enterprise adoption — users don't have to abandon their existing tools.

Sadra's reaction: "this is gonna work with existing applications" — Option 2 is safer. The wiki KB already partially does this: Quartz renders the wiki in a browser (existing web infrastructure), not inside a proprietary tool. The question is whether to go further into native application integration.

## Open Questions

- Does embedded views require API access to the target application? (For Excel: Office.js; for Word: Word API)
- Can the "edit mode / render mode" distinction be surfaced gracefully without disrupting normal document editing?
- How does provenance work when a view is embedded in a document the user may edit manually?
