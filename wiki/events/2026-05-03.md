## First Live Demo of the Wiki to Sumit

Sumit saw the wiki KB live for the first time and engaged deeply with its architecture.

### Semantic Index Framing

Sumit reached the key insight: the wiki is not a view output — it is a **semantic index**, a structured intermediate that makes future views efficient. "Wiki is providing the right kinds of artifacts to do good indexing about different kinds of entities, such as people, topics." Vector indexes (the standard agentic approach) are mechanical; the wiki index is semantic and concept-typed.

This crystallized the [[information-knowledge-pipeline|Raw → Information → Knowledge → Views pipeline]]: raw data is information; the wiki is knowledge; views are purpose-specific renderings on top of knowledge. Views do NOT re-read raw data — they operate on the wiki. This is the efficiency argument.

### Wiki Schema is Defined by Ingest

Sumit observed that the ingest command implicitly defines the schema of the wiki: "your ingest is in some sense describing the structure of this wiki." The choice of people / concepts / events / projects as entity types is not ontologically fixed — it's a design decision embedded in the ingest prompt. New entity types (e.g., "speaker style") can be added and future ingests will extract them retroactively.

### Embedded Views (Option 2)

Sumit proposed two options for how views get rendered:

- **Option 1** (current) — Claude Code CLI generates complete views from scratch. "All applications are dead."
- **Option 2** — Views are embedded in existing applications (Word, PowerPoint, Excel) as prompt-powered "holes" in a document template. Each cell or section has an associated prompt; the application is the container. See [[embedded-views]].

Sadra favored Option 2 as safer for positioning ("this is gonna work with existing applications").

### Vibe Session as View Definition UX

Sumit flagged a UX gap: defining the right view requires iteration. "You need to give me a facility to do a vibe session in the process of me defining that right view." After the session, the user should be able to distill it into a reusable view definition and new skills. This is the natural creative process that the CLI currently skips over.

### View Prompt Transparency

Sumit asked how users see the prompt underlying a rendered view. Currently it's in the `.claude/commands/` directory — not surfaced in the rendered wiki. Ideal: right-click a view to see its source prompt. This is a UX gap for the graphical interface.

### Karpathy Connection Formalized

Sumit learned about Karpathy's LLM-wiki post in detail for the first time. Sadra explained it. Sumit confirmed the key differentiator: "We never call it a personal knowledge base." The [[wiki-kb-project|Wiki KB Project]]'s novelty over Karpathy is the team/organizational scope — [[semi-private-mashing|semi-private mashing]] is the mechanism that makes this possible. See [[llm-wiki-landscape]].

### Sumit's Own Use Case

Sumit wants to run the system on his private Microsoft meeting transcripts. Sadra explained: clone the repo, open Claude Code, put raw data in `raw/`, run ingest. Sumit plans to try it on his own conversations with colleagues.

Separately, Sadra is already applying the system to his cousin's chemistry PhD emails — stress-testing scale beyond the toy examples used in the research project.
