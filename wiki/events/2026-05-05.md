# May 5, 2026 — Two Syncs: AI Slop Diagnosis + Full Terminology Overhaul

Two back-to-back meetings between [[sadra-sabouri|Sadra]] and [[sumit-gulwani|Sumit]]: a 15-minute briefing and a 1h11m deep session.

## Session 1 (15 min) — AI Slop and the Case for Skills

Sumit described a painful experience trying to generate his "connect" (performance review) using AI. Despite providing: connects of all direct reports, meeting transcripts, past examples, blog posts, accolades, paper lists, and hand-written guidance paragraphs — the output was "slop." The model produced boilerplate, ignored his specific notes, and then when challenged, claimed to have found the right material in the transcript anyway.

His diagnosis: **"Having raw sources and a processing pipeline is not good enough. There is something more."** What's missing is the skill — the creative specification of exactly how to compute the output. Prompts need to encode methodology, not just intent. This is not a retrieval problem; it is a computation problem.

He also requested 3 concrete wiki examples for different audiences to accompany his Thursday post: a team building an agent (Athena), a writer/knowledge worker, and a software developer.

The terminology shift crystallized here: "This is what I've been calling a workflow… maybe just call it a skill now." See [[skills]].

## Session 2 (1h11m) — Terminology Overhaul and Thursday Prep

### Precise Definitions Settled

The most precise definitions of core concepts to date, per [[T14-2026-05-05-2]]:

- **View** — "A view is a template. Its holes are filled by executing prompts against an optional specification of input sources. Each prompt can optionally be associated with a skill."
- **Workflow** (narrowed) — "A representation of the data dependencies between views and the raw sources. When a raw source updates, the workflow tells us which prompt needs to rerun and in what order." Roughly analogous to GitHub Actions.
- **Skill** — what was previously called "creative workflow": the constructive specification of HOW to compute one prompt's hole. A gallery of skills is a community library of knowledge distillation methods.

The rename was confirmed: "We switched from workflows to using skills." [[workflows]] is now narrowly the data-dependency graph; [[skills]] is the creative computation method per prompt.

### Context Base vs. Knowledge Base

Sumit shared an "applied compute" paper he'd been reading — it calls the same architecture a **context base** (for agents), with three phases: Remember (ingest), Refine (curation agent), Retrieve (agent runtime). This project builds a **knowledge base** for *human* knowledge work, not an agent automation pipeline. The distinctions:

- Context base goal: agents complete long-horizon cross-application tasks autonomously
- Knowledge base goal: humans define views and consume synthesized knowledge; agents are tools, humans are in the loop

Karpathy subsumes the Second Brain / CODE framework in the AI era. Applied compute extends this for agents. This project extends it for human-centered knowledge work with views, scalable organizational scope, and semi-private mashup. See [[context-base]].

### Three Anchors for "Living Documents"

The living-document concept got three concrete anchor examples from software and productivity tools:
1. **Dashboards** — auto-update when new data arrives
2. **Excel formulas / recalc engine** — static cells vs. formula cells; the formula recalcs when inputs change
3. **Excel "Equals Copilot" feature** — prompt-powered cells that apply the prompt when inputs change

The key insight: previous tools (dashboards, formulas) only automated *objective*, code-expressible computation. The knowledge base enables the same recalc model for *subjective* and *aesthetic* content — because LLMs can now handle that.

### Document as View

Every paragraph of a document can be a prompt. Sumit articulated this while describing his internal post: the motivating story section, the definitions section, the applications section — each could be a prompt over the KB, populated by a skill. If done this way, the post is itself a view, and would auto-update as new transcripts arrive. "I want this document to be a view."

### Wiki Maintenance: PR Model

Who updates the wiki when a concept is deprecated? Sumit's answer: open-source evolution. People depend on it, so they are incentivized to keep it current. Agents submit PRs; humans review. The same PR/review model used for code now applies to knowledge. "Even today, code PRs are being submitted using agents."
