## Live Views

A **[[views|view]]** is a document where every section is a prompt executed against the knowledge base — not a paragraph someone wrote. When new content arrives, the prompts rerun. The document is always current. *[[views|Document as View]]*

- **This deck** — generated by running `/presentation` over the wiki. A new meeting is ingested; the deck is regenerated; it reflects what changed. No manual rewrite. This is the [[living-system|Excel recalc model]] applied to subjective content: a formula cell recomputes when `A1` changes; a view recomputes when a transcript is added. *[[living-system]]*
- **Teams auto-summary → query view** — Teams generates one fixed post-meeting summary template. A live view extends this to user-definable queries run on demand across all meetings: "give me contradictions between two people." That query has no fixed-template answer; it requires a prompt over the whole corpus. *Sumit Gulwani — [[Transcripts]]*
- **Sumit's connect as a view** — every section of the performance review (motivating story, definitions, applications) is a prompt guided by a [[skills|skill]] that encodes *how* to prioritize and synthesize, not just what to retrieve. When a new quarter's 1:1 transcripts are ingested, the review updates. The skill is precisely what was missing from the three-day attempt. *[[Sadra-5May2026-TRANSCRIPT2]]*

The Excel analogy is precise: a formula cell is a saved computation, not a stored value. A view is a saved prompt, not a stored document. *[[living-system]]*

## Semi-Private Mashing

The organizational brain only works if individuals can contribute personal knowledge safely. Today, that filter does not exist — and the failure mode is already documented.

Sumit's agent was given access to a draft he was helping a colleague with. It pulled **three sentences of her unpublished work** into his output. He caught it manually; she asked him to remove it. No system flagged it. This is what happens without a structural privacy boundary: private content leaks at the agent's discretion, not the owner's. *Sumit Gulwani — [[Sadra-5May2026-TRANSCRIPT2]]*

**[[semi-private-mashing|Semi-private mashing]]** is the filter. A 1:1 transcript contains project decisions (shareable), personal context (not shareable), and half-formed ideas (judgment call). The filter is itself a [[views|view]] — a declarative transformation applied to the private source *before* anything enters the shared layer. Raw private sources never enter the shared KB; only filtered, distilled output does. *[[semi-private-mashing]]*

The concrete mechanism is a **PR model**: a user runs a private KB locally; they push only selected, distilled knowledge units to the shared org KB via a pull request. The PR boundary is structural — the unpublished-work incident becomes impossible by construction, not by the user catching it manually. *[[local-first-kb|Local-First KB]] — [[Personal Private Knowledge Management]]*

Concretely: applying [[skills|grounded theory analysis]] (inductive coding → clustering → naming) as a skill to a private 1:1 transcript extracts concept clusters — "views", "privacy concerns" — without exposing the raw conversation. Only the thematic map enters the shared KB. The filter is a methodological transformation, not a redaction rule. *[[Sadra-5May2026-TRANSCRIPT2]]*

**This wiki was built from private 1:1 transcripts via exactly this mechanism.** Every claim on every wiki page is a filtered synthesis from a private source — not a raw quote.

> speaker note: The unpublished-work incident is the most visceral example for this audience. "No system caught it" is the key line. The PR model inverts this: leakage is structurally impossible, not manually prevented.

## What to Build Next

1. **Percolation strategy for live recalc** — when a raw source updates, which views must rerun, in what order, at what cost? Today's wiki recomputes blindly on every regeneration. *Why it matters:* the [[living-system]] promise depends on this being efficient at scale — 1,000 sources and 50 view templates cannot all rerun on every ingest. *First step:* instrument the existing DAG, measure recompute cost on a 1,000-source corpus, design selective propagation based on source-to-view dependency tracking.

2. **Semi-private enforcement at the agent level** — how do you *guarantee* (not merely hope) that private content stays out of the shared layer when an agent is the filter? *Why it matters:* the unpublished-work incident is the failure mode every enterprise deployment will hit; manual review does not scale. *First step:* design a declarative privacy policy language + audit log; pilot on 1:1 transcripts with synthetically injected violations; measure detection rate. *[[semi-private-mashing]]*

3. **AI-reviewed PRs for the org KB** — when a personal KB contributes upstream, what reviews the PR for contradictions, leakage, and provenance? *Why it matters:* without this, the [[kb-as-codebase|KB-as-codebase]] PR model collapses to "human review at scale" and adoption fails. *First step:* prototype a reviewer agent on three contribution types (concept addition, claim revision, source supersede); measure precision/recall on injected violations. *[[local-first-kb]]*

Each answer to these questions improves this deck on the next regeneration: percolation makes it cheaper to regenerate, enforcement lets it safely cite private 1:1s, and AI-reviewed PRs let collaborators contribute knowledge that propagates here automatically.
